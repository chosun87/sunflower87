## 📌 2단계. 매매 내역 (매매 히스토리) 다중 검색 기능 구현
- **대상 API:** `GET /api/transactions`
- **상세 요구사항:**
  - 프런트엔드에서 상단 검색 필터를 통해 요청할 수 있도록 아래 쿼리 파라미터를 추가하십시오.
    - `account_id` (계좌 식별자, Optional)
    - `stock_code` (종목코드, Optional)
    - `start_date` / `end_date` (조회 기간, YYYY-MM-DD, Optional)
  - SQLAlchemy를 활용하여 파라미터가 존재할 때만 dynamic `.filter()`를 적용하는 동적 쿼리를 작성하십시오.
  - 파라미터가 누락된 경우 기존의 전체 매매 내역  반환 정책을 유지하되, 무결한 연대기적 정렬(Chronological Order) 상태를 보장하여 송출하십시오.

## 📌 3단계. 계좌별 보유 종목 API 확장 (0주 보유 포함)
- **대상 API:** `GET /api/stocks/holdings` (혹은 자산 잔고 조회 담당 API)
- **상세 요구사항:**
  - 기존에는 현재 보유 수량이 1주 이상인 종목만 잔고 테이블(`stocks`)에서 조회하여 반환했으나, 이제는 **과거 매매 이력이 존재하여 현재 보유 수량이 0인 종목**도 배열에 포함하여 송출해야 합니다.
  - 이를 통해 프런트엔드가 '전체 종목 조회' 모드일 때 0주 보유 종목을 화면에 마킹할 수 있도록 지원하십시오.
  - 응답 데이터의 각 오브젝트에 `quantity: 0` 정보가 정확히 담겨야 합니다.

---

## 📌 4단계. 캔들 차트 기간 이동(Pan) 대응 데이터 버퍼 서빙
- **대상 API:** `GET /api/stocks/ohlcv`
- **상세 요구사항:**
  - 프런트엔드 차트의 좌우 드래그(Pan) 및 스크롤 인터랙션을 지원하기 위해 조회 기간 파라미터(`start_date`, `end_date`)를 확장 수용하십시오.
  - **🚨 [초핵심] 80-Day Buffer 규격 준수:** - 프런트엔드 단에서 화면 프리징 없이 즉각적으로 20일 이동평균선(20MA) 및 5일 이동평균선(5MA)을 연산할 수 있어야 합니다.
    - 따라서, 요청받은 `start_date` 시점보다 **최소 80~100거래일 앞선 과거 시계열 데이터부터 누적하여 오름차순(ASC)으로 정렬**한 뒤 페이로드 선두에 결합(Prepend)하여 송출하십시오.
  - 데이터의 빈틈을 메우는 Gap Backfill 알고리즘 무결성을 유지하십시오.
