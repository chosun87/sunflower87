## 🚨 0순위. [긴급 보완] P1 코드베이스 누락 건 보완 (Pydantic 무결성 검증)
- **대상:** `be/schemas.py`의 `TransactionCreate` 스키마

## 🛠️ 0.5순위. 종목 마스터(cache_stocks) ETF 통합 및 Soft Delete 패턴 도입
- **내용:** 상장폐지/코드변경 종목 관리 부재 이슈 해결 및 ETF 종목 검색 누락 패치를 단일 스키마 개선 작업으로 통합 전개합니다.
- **상세 요구사항:**
  1. **DB 스키마 확장:** `cache_stocks` 테이블에 상장 상태를 추적할 수 있는 `is_active` (BOOLEAN, 기본값 True) 컬럼을 추가하십시오.
  2. **Lifespan 시딩 로직 고도화 (ETF 통합 및 동적 Diff):**
     - 서버 기동 시 기존 일반 주식 수집 로직에 `pykrx.stock.get_etf_ticker_list(date)`를 결합하여 ETF 종목까지 함께 확보하십시오. 병합 시 ETF 종목의 `market` 컬럼에는 'ETF'가 기입되도록 처리합니다.
     - 당일 수집된 외부 정상 리스트와 로컬 DB 리스트를 대조(Diff)하여, **과거 매매 이력 등으로 로컬 DB에는 존재하지만 오늘 상장 상태가 확인되지 않는(상장폐지 등) 종목은 `is_active = False`로 변경(Soft Delete)** 하십시오. (과거 자산 역산 무결성을 위해 물리적 데이터 삭제는 절대 금지합니다.)
  3. **검색 API 조건 제한:** 프런트엔드 종목 검색창에서 호출하는 자동완성/조회 API는 반드시 `is_active == True` 조건이 충족된 종목만 반환하도록 한정하십시오. (단, 과거 매매 히스토리 조회를 위한 API 호출 시에는 상태와 무관하게 종목명이 올바르게 표출되어야 합니다.)

## 📌 2단계. 거래 내역(매매 히스토리) 다중 검색 기능 구현
- **대상 API:** `GET /api/transactions`
- **상세 요구사항:**
  - 프런트엔드 요구 파라미터 수용: `account_id` (Optional), `stock_code` (Optional), `start_date` / `end_date` (Optional)
  - 최근 리팩토링 구조를 엄수하여, 복잡한 DB 동적 필터링 로직은 라우터가 아닌 분리된 **`be/services/transaction_service.py` 서비스 레이어 내부**에 구현하고 라우터에서는 이를 호출만 하도록 처리하십시오.
  - 파라미터가 누락된 경우 기존의 전체 거래 내역 반환 및 연대기적 정렬(Chronological Order) 상태를 완벽하게 유지하십시오.

## 📌 4단계. 캔들 차트 기간 이동(Pan) 대응 데이터 버퍼 서빙
- **대상 API:** `GET /api/stocks/ohlcv`
- **상세 요구사항:**
  - 차트의 좌우 드래그 조작을 위한 조회 기간 파라미터(`start_date`, `end_date`)를 수용하십시오.
  - **80-Day Buffer & 자가치유 보강:** 요청받은 `start_date` 기준 최소 80~100거래일 앞선 과거 시계열 데이터부터 누적하여 오름차순(ASC)으로 정렬 정송해야 합니다. 이를 위해 주말/휴장일 여유 버퍼를 감안하여 날짜 범위를 동적으로 계산(`target_start_date = start_date - 120일`)한 뒤 DB 조회를 수행하십시오.
  - 지정된 `start_date` 보다 더 이전의 데이터가 DB에 부족한 경우, `portfolio_service.py` 내 `sync_ohlcv_cache` 가 외부 `pykrx`를 호출하여 캐시 테이블 앞단을 유동적으로 채워 넣는 **Historical Backfill 자가 치유 알고리즘**을 완성하십시오.
