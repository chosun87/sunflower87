# P-6-01-3BE-설계 의견 (계좌 관리 기능)

## 1. REST API 설계 (계좌 관리 CRUD)
DB 파트의 의견을 수렴하여 `account` 테이블에 대한 기본 CRUD API를 구성합니다.

### 1.1 계좌 목록 조회
* **Method & URI**: `GET /api/accounts/`
* **Response**: 등록된 계좌 목록 (배열). (단, `dt_deleted`가 null인 항목만 조회)
* **정렬**: `acc_order` ASC, 생성일 오름차순 기준.

### 1.2 계좌 등록
* **Method & URI**: `POST /api/accounts/`
* **Request Body**:
  ```json
  {
    "acc_cd": "string (고유코드)",
    "acc_nm": "string",
    "acc_company_nm": "string"
  }
  ```
* **Validation**: 이미 존재하는 `acc_cd`인지 중복 검사 (중복 시 400 Error).

### 1.3 계좌 정보 수정
* **Method & URI**: `PUT /api/accounts/{acc_cd}`
* **Request Body**:
  ```json
  {
    "acc_nm": "string",
    "acc_company_nm": "string"
  }
  ```
* **로직**: `acc_cd`는 변경 불가. 이름 및 금융사명만 업데이트.

### 1.4 계좌 삭제 (소프트 삭제)
* **Method & URI**: `DELETE /api/accounts/{acc_cd}`
* **로직**: 데이터베이스에서 행을 지우지 않고 `dt_deleted = CURRENT_TIMESTAMP`로 업데이트.
* **추가 고려**: 기존 거래 내역(`Transaction`)이 존재해도 삭제(비활성)는 가능하도록 허용합니다. (잔고 동기화 시에는 제외되거나 히스토리용으로만 참조)

## 2. 작업 계획
1. `app/schemas.py`에 계좌 조회/생성/수정용 Pydantic 모델(`AccountCreate`, `AccountUpdate`, `AccountResponse` 등) 작성.
2. `app/routers/accounts.py`에 상기 엔드포인트 구현 (필요시 `services/account_service.py` 분리).
3. Swagger UI를 통한 API 정상 작동 검증.
