# Phase 5 기획서 (PL) - 일일 잔고 동기화 기획 (수정본)

## 1. 개요
현재 DB에 존재하는 `account_balance_daily` (계좌별 일자별 잔고) 테이블의 운영 정책과 일일 잔고 동기화 기능에 대한 기획 및 요구사항을 새롭게 정의합니다.

## 2. 사용자 문의 사항에 대한 기획(PL) 의견

### Q1. API 엔드포인트 변경 제안
> **문의:** 일일 잔고 동기화(/dataManagement)로 수정이 되는 테이블이 account_balance_daily 하나라면 네이밍을 변경하는게 어떨지.
* **기획 의견:** 적극 찬성합니다. 현재 백엔드에는 이미 `/api/accounts/sync_balance_daily` 와 같은 엔드포인트가 준비되어 있습니다. 
* 프론트엔드의 메뉴 라우팅 이름(`/dataManagement`)은 너무 포괄적이므로, 백엔드의 목적과 통일성 있게 **`/sync_balance_daily`** (현재 메뉴 key와 동일) 또는 **`/recalc_account_balance`**로 변경하여 직관성을 높이는 것을 제안합니다.

### Q2. 재계산 기간 설정 제안
> **문의:** 일일 잔고 동기화 실행하면 투자 시작일 부터 어제일자까지 재계산을 하는 것 맞나? 기간을 입력 받아서 그 기간만을 재계산하는 방법은 어떨까? 기본 기간은 1개월.
* **기획 의견:** 매우 훌륭한 제안입니다. `account_balance_daily`에 데이터가 쌓일수록 '투자 시작일'부터 매번 전체를 재계산하는 것은 서버/DB 리소스 낭비가 큽니다.
* **설계 방안:** 
  - 백엔드 재계산 API에 `start_date`와 `end_date` 파라미터를 추가합니다.
  - UI에서는 기본적으로 **최근 1개월(예: 한 달 전 ~ 어제)**로 기간이 자동 세팅되게 하고, 달력(Date Picker)을 통해 특정 기간만 동기화할 수 있도록 지원합니다.

### Q3. 오늘자 레코드 처리 및 실시간 동기화
> **문의:** account_balance_daily에는 오늘자 레코드가 있고, 실시간으로 변경이 되는건지? 오늘자 레코드로 pki가 계산되는 건지?
* **기획 의견:** 
  - `account_balance_daily`와 같은 테이블은 원칙적으로 **일 마감(EOD, End of Day) 기준의 확정 데이터 스냅샷**을 보관하는 용도여야 합니다.
  - **오늘자 데이터 실시간 변경 불가:** 장중(오늘) 실시간 체결이 일어날 때마다 이 테이블을 업데이트하면 트랜잭션 부하가 큽니다. 따라서 이 테이블에는 **'어제'까지의 마감 데이터만 기록**합니다.
  - **당일 실시간 PKI 계산:** '현재(오늘)' 시점의 수익률과 잔고는 실시간 계좌 정보(`account` 테이블의 현금 잔고)와 현재 주식 보유 정보(`stock` 테이블)를 합산하여 계산 및 조회되도록 분리 운영하는 것이 최적의 아키텍처입니다.

### Q4. 계좌개설일 기반 잔고 계산 반영
> **문의:** account_balance_daily 계산시에 각 계좌의 계좌개설일 이후부터 계산할 수 있나요?
* **기획 의견:** 좋은 의견입니다. `account` 테이블에 `dt_opened` (계좌개설일) 컬럼을 추가하고, 일일 잔고 동기화 시 이 날짜를 최소 시작일(Bound)로 설정하여 그 이전의 불필요한 기간에 대해 연산하지 않도록 방어 로직을 추가하겠습니다.
