# Phase 5: 일일 잔고 동기화 기획 구현 계획 (Implementation Plan)

본 계획서는 `docs/Phase-5/` 경로에 작성된 기획(PL), 백엔드(BE), 프론트엔드(FE), 데이터베이스(DB), 테스터(QA) 파트의 의견을 취합하여 시스템에 반영하기 위한 실행 계획입니다.

## 1. User Review Required

> [!IMPORTANT]
> **엔드포인트 명칭 변경 및 UI 라우트 변경**
> 기획안(P-5-PL.md)에 따라 프론트엔드의 기존 `DataManagement` 페이지를 `SyncDailyBalance` 페이지로 역할을 축소/명확화하고 라우트 이름을 `/sync_balance_daily`로 변경합니다. (추후 다른 데이터 관리 기능이 필요해지면 별도로 추가)

> [!WARNING]
> **DB 트랜잭션 주의사항**
> DB 파트(P-5-DB.md) 요구사항에 따라, 재계산 시 입력받은 `start_date` ~ `end_date` 구간의 기존 `account_balance_daily` 데이터를 먼저 `DELETE`한 뒤 다시 갱신(`INSERT`/`UPSERT`)하도록 로직을 구현합니다. 당일(오늘) 날짜에 대한 잔고 갱신은 방어 로직으로 원천 차단됩니다.

## 2. Proposed Changes

### Backend (BE)

#### [MODIFY] [database.py](file:///c:/01_Projects/sunflower87/be/database.py)
* `Account` 모델에 `dt_opened` (계좌개설일) 컬럼을 `VARCHAR` 타입으로 추가하여 해당 일자부터 잔고를 계산할 수 있는 기준점을 마련.

#### [MODIFY] [schemas.py](file:///c:/01_Projects/sunflower87/be/schemas.py)
* `SyncAccountDailyBalanceRequest` 스키마 추가
  * `start_date`: Optional[str] (YYYY-MM-DD)
  * `end_date`: Optional[str] (YYYY-MM-DD)

#### [MODIFY] [routers/account.py](file:///c:/01_Projects/sunflower87/be/routers/account.py)
* `POST /api/accounts/sync_balance_daily` 및 `POST /api/accounts/{acc_cd}/sync_balance_daily` 에 Payload(`SyncAccountDailyBalanceRequest`)를 선택적으로 받도록 스키마 연동.
* 서비스 레이어 함수 호출 시 인자로 `start_date`, `end_date` 전달.

#### [MODIFY] [services/account_balance_daily_service.py](file:///c:/01_Projects/sunflower87/be/services/account_balance_daily_service.py)
* `sync_account_balance_daily` 함수에 `req_start_date`, `req_end_date` 파라미터 추가. (기존 `sync_account_balance_daily` 함수명 변경)
* 파라미터 누락 시, 기본값은 `1개월 전` ~ `어제` 로 자동 설정. 단, 해당 계좌의 `dt_opened`(계좌개설일) 이전으로 넘어가지 못하도록 하한(Bound) 방어 로직 추가.
* 미래 날짜 및 '오늘' 날짜를 `end_date`로 지정 시, 강제로 '어제'로 조정(방어 로직). 실제 데이터 생성은 주식 시장 거래일(Trading Day) 기준으로만 생성되므로 휴일은 자연스럽게 건너뜀.
* DB 파트 요구사항: 재계산을 수행하기 전, `[start_date, end_date]` 범위 내의 `AccountDailyBalance` 레코드를 해당 계좌(`acc_cd`)에 대해 일괄 `DELETE`.
* 재계산을 위해 거래 내역(`Transaction`, `TransactionCash`)을 초기부터 순회하되, `AccountDailyBalance`에 저장(`INSERT`)하는 대상은 `start_date` 이후로 한정하여 최적화 및 정합성 유지.
