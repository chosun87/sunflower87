# Phase 5 데이터베이스 설계 의견 (DB)

## 1. 기획안에 대한 DB 파트 의견
테이블명을 `account_daily_balance`로 정정해주셔서 감사합니다. 이미 `database.py`에 아래와 같이 구성되어 있음을 확인했습니다.
```python
class AccountDailyBalance(Base):
    __tablename__ = "account_daily_balance"
    acc_cd = Column(String, ForeignKey("account.acc_cd"), primary_key=True)
    trade_date = Column(String, primary_key=True)  # YYYY-MM-DD
    cash_balance = Column(Integer, nullable=False, default=0)
    stock_eval_balance = Column(Integer, nullable=False, default=0)
    total_balance = Column(Integer, nullable=False, default=0)
    return_rate = Column(Float, nullable=False, default=0.0)
```
해당 구조는 기간별 총 자산 추이 및 수익률(Return Rate)을 관리하기에 최적화되어 있습니다. 기획안대로 일 마감(EOD) 전용으로 활용하는 것에 100% 동의합니다.

## 2. 재계산(Recalculate) 시 데이터 처리 전략 (기간 설정 지원)
* 프론트엔드로부터 특정 기간(`start_date` ~ `end_date`) 재계산 요청이 인입될 경우, 
* DB 상에서는 해당 `acc_cd`의 `trade_date`가 범위 내에 있는 기존 레코드를 먼저 `DELETE` 하고, 재계산된 값을 `INSERT` 하는 방식이 가장 깔끔합니다.
* 추가로, `account` 테이블에 `dt_opened` (계좌개설일) 컬럼을 `VARCHAR` 타입으로 추가하여 일일 잔고 동기화 시 불필요한 과거 데이터 연산을 방어할 수 있도록 스키마를 업데이트합니다.
* 오늘 날짜(`trade_date = 오늘`)에 대한 쿼리 삽입은 원천적으로 제한하여, 실시간 데이터와의 혼용으로 인한 무결성 훼손을 방지합니다.
