# P-6-01-1PL: 계좌 관리 기능 기획안

## 1. 개요
* **기획 목적**: 사용자가 자신의 계좌를 등록, 조회, 수정, 삭제할 수 있는 '계좌 관리 기능'을 제공하여, 전체 자산 및 일일 잔고 동기화의 기반 데이터를 마련한다.
* **대상 시스템**: 프론트엔드(React) UI 및 백엔드(Python/FastAPI 등) API, 데이터베이스(SQLite)

## 2. 주요 기능 및 요구사항

### 2.1 계좌 목록 조회
* **기능 설명**: 등록된 계좌의 전체 목록을 조회하여 화면에 표시한다.
* **표시 항목**: 계좌코드(acc_cd), 계좌명, 금융사명, 계좌 상태(활성/비활성), 잔고(선택)
* **UI 컴포넌트**: PrimeReact의 `DataTable`을 사용하여 구현하며, 페이징 및 정렬 기능을 제공한다.

### 2.2 계좌 등록
* **기능 설명**: 새로운 계좌 정보를 시스템에 등록한다.
* **입력 항목**: 계좌코드(필수, 중복 불가), 계좌명(필수), 금융사명(선택), 계좌 상태(기본값: 활성)
* **UI 컴포넌트**: PrimeReact의 `Dialog`와 `InputText`, `Dropdown` 등을 조합하여 모달 형태로 제공한다.

### 2.3 계좌 수정
* **기능 설명**: 기존에 등록된 계좌의 정보(계좌명, 금융사명, 상태 등)를 수정한다.
* **제약 사항**: 계좌코드(`acc_cd`)는 고유 식별자이므로 수정할 수 없다.

### 2.4 계좌 삭제
* **기능 설명**: 더 이상 사용하지 않거나 잘못 등록된 계좌를 삭제한다.
* **제약 사항**: 해당 계좌와 연동된 거래 내역이나 일일 잔고 데이터가 존재할 경우, 삭제 대신 '비활성' 처리하거나 사용자에게 경고 후 연관 데이터를 함께 삭제할지 결정해야 한다. (소프트 삭제 권장)

## 3. UI/UX 화면 설계서 (간이)
* **메뉴 위치**: 사이드바 (Sidebar) > '계좌 관리' 메뉴 추가
* **UI/UX 디자인 원칙**: 기존 **주식 매매 내역** 및 **계좌 입출금 내역** 화면과 동일한 레이아웃 및 UX 패턴 적용
* **메인 화면 (계좌 목록)**:
  * 상단 툴바(Toolbar): [새 계좌 등록] 버튼 및 **[삭제 계좌 포함] 체크박스** 배치, (필요시) [선택 삭제/엑셀 다운로드] 등
  * 중앙: 계좌 목록 DataTable (기존 내역 화면과 동일한 테마, 필터, 페이징, 정렬 스타일 적용)
  * 등록 및 수정: 기존 내역 등록/수정과 동일한 스타일의 Dialog(모달) 또는 인라인 폼 제공 (각 행의 끝에 [수정], [삭제] 액션 아이콘 배치)

## 4. 향후 작업 계획
* **DB**: `accounts` 테이블 스키마 설계 및 검토
* **BE**: CRUD API 규격 정의 및 엔드포인트 구현
* **FE**: PrimeReact 기반의 UI 컴포넌트 개발 및 API 연동
* **QA**: 기능 검증을 위한 테스트 케이스(TC) 작성

## 5. 개발자 의견 (DB, BE, FE, QA 파트 피드백 및 테스트 계획)

> 각 파트 담당자(DB, BE, FE, QA)의 세부 설계 및 테스트 계획입니다.

### DB 담당자 (DB전문가_SQLite) 의견:
* **스키마 적합성**: 기존 `account` 테이블 스키마를 그대로 수용 가능. (계좌코드, 계좌명, 금융사명, 상태 등)
* **상태 관리**: 상태(활성/비활성)는 `dt_deleted` 컬럼의 Null 여부로 판단하는 **소프트 삭제(Soft Delete)** 방식을 적용.
* **무결성 유지**: 계좌 삭제 요청 시 `DELETE` 대신 `UPDATE account SET dt_deleted = CURRENT_TIMESTAMP WHERE acc_cd = ?` 방식으로 처리하여 `Transaction` 등 연관 테이블의 무결성 보호.

### BE 담당자 (BE개발자_Python) 의견:
* **계좌 목록 조회**: `GET /api/accounts/?include_deleted=false` (쿼리 파라미터 `include_deleted` 추가. `true`일 경우 `dt_deleted`가 null이 아닌 항목도 함께 조회, `acc_order` 및 생성일 기준 정렬)
* **계좌 등록**: `POST /api/accounts/` (중복 `acc_cd` 체크 및 400 에러 처리)
* **계좌 정보 수정**: `PUT /api/accounts/{acc_cd}` (계좌코드는 변경 불가, 이름/금융사명 업데이트)
* **계좌 삭제**: `DELETE /api/accounts/{acc_cd}` (소프트 삭제 적용)
* `app/schemas.py` 및 `app/routers/accounts.py` 에 API 엔드포인트 구현 예정.

### FE 담당자 (FE개발자_React) 의견:
* **레이아웃**: 기존 내역 페이지와 동일한 `Card`, `DataTable` 프레임 사용 (페이징, 정렬 포함)
* **상단 툴바**: [새 계좌 등록] 버튼과 함께 **[삭제 계좌 포함] Checkbox** 배치. 체크 상태 변경 시 API 재호출하여 목록 갱신.
* **등록/수정**: 기존 `Dialog` 기반 모달 재사용 (`InputText`, `Dropdown` 조합). 폼 작성 후 저장 시, `dialogUtils.ts`의 `showConfirm()`(confirmDialog)을 띄워 "등록(수정)하시겠습니까?"라고 다시 한 번 확인받음. API 완료 및 오류 발생 시 `showNotice()` 알림 적용.
* **삭제**: 삭제 클릭 시 PrimeReact `confirmDialog`를 통해 확인 후 처리 (`/accounts` 페이지 갱신)
* `src/pages/AccountManagement.tsx` 신규 생성 및 `SidebarData.ts`에 메뉴 추가 예정.

### QA 담당자 (QA테스터) 테스트 계획:
* **TC-01. 조회**: 기존 등록된 계좌 목록이 데이터 테이블에 정상적으로 페이징/정렬되어 표시되는지 확인.
* **TC-02. 등록**: 정보 입력 후 저장 클릭 시 "등록하시겠습니까?" 컨펌창이 나타나며, 승인 시 정상 등록되고 `showNotice()` 성공 알림과 함께 목록이 갱신되는지 확인.
* **TC-03. 중복 예외**: 기 등록된 계좌코드로 등록 시 400 에러 및 `showNotice()` 경고 알림이 표시되는지 확인.
* **TC-04. 수정**: 계좌코드 제외 정보(이름, 금융사) 수정 시 DB 반영 및 목록 갱신 확인.
* **TC-05. 삭제**: 삭제 버튼 클릭 후 확인 시 목록에서 사라지고 DB에서는 `dt_deleted`만 업데이트(소프트 삭제) 되는지 검증.
* **TC-06. 삭제 계좌 포함 조회**: [삭제 계좌 포함] 체크박스를 체크했을 때 삭제 처리된 계좌가 목록에 나타나고, 체크 해제 시 다시 사라지는지 확인.
