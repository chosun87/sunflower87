# P-6-01-5QA-테스트 계획 (계좌 관리 기능)

## 1. 테스트 목적
* 계좌의 조회, 생성, 수정, 삭제(소프트 삭제) 기능이 정상적으로 동작하는지 확인합니다.
* 예외 상황(중복된 계좌코드 입력, 필수 항목 누락 등)에서 적절한 에러 메시지가 반환되는지 확인합니다.
* 계좌 삭제 시, 연동된 타 테이블(`Transaction` 등)에 물리적인 제약 조건 오류가 발생하지 않는지 검증합니다.

## 2. 테스트 케이스 (TC)

### TC-01. 계좌 목록 조회
* **사전 조건**: DB에 계좌 데이터가 1개 이상 존재.
* **테스트 절차**: FE에서 계좌 목록 페이지로 이동.
* **예상 결과**: 등록된 계좌 목록이 데이터 테이블에 정상적으로 표시되며 페이징 및 정렬이 정상 작동해야 함.

### TC-02. 정상적인 계좌 등록
* **사전 조건**: 새로운 계좌코드 준비 (예: `ACC-TEST-001`).
* **테스트 절차**: [새 계좌 등록] 모달에서 필수 항목 입력 후 저장.
* **예상 결과**: 등록 성공 알림(Toast)이 발생하고, 테이블이 갱신되어 새로 등록한 계좌가 표시됨.

### TC-03. 중복된 계좌코드로 등록 시도
* **사전 조건**: 이미 등록된 계좌코드로 등록 시도.
* **테스트 절차**: 동일한 계좌코드를 입력하고 저장.
* **예상 결과**: BE에서 400 에러 반환 및 FE에서 "이미 존재하는 계좌코드입니다"와 같은 경고 토스트 표시.

### TC-04. 계좌 정보 수정
* **사전 조건**: 등록된 계좌 선택.
* **테스트 절차**: 수정 버튼 클릭 -> 계좌명과 금융사명을 변경 후 저장.
* **예상 결과**: 변경된 정보가 DB에 반영되고, 화면의 계좌 목록도 즉시 갱신됨. (계좌코드는 수정 불가한 상태여야 함)

### TC-05. 계좌 삭제 (소프트 삭제)
* **사전 조건**: 삭제할 계좌 존재.
* **테스트 절차**: 삭제 버튼 클릭 -> Confirm Dialog 승인.
* **예상 결과**: 화면(목록)에서 해당 계좌가 사라짐. DB 확인 시 `account` 테이블에서 레코드가 물리적으로 지워지지 않고 `dt_deleted`에 현재 시각이 기록되어야 함.

## 3. 테스트 환경
* **Backend**: Swagger UI (`/docs`)를 통한 API 직접 테스트
* **Frontend**: 로컬 개발 서버 (`http://localhost:5173/accounts`)에서 수동 테스트
