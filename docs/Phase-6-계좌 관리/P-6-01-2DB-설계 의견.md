# P-6-01-2DB-설계 의견 (계좌 관리 기능)

## 1. 기존 DB 스키마 분석
현재 `database.py`에 정의된 `Account` 모델은 다음과 같습니다.
* `acc_cd` (String, PK): 계좌코드
* `acc_nm` (String, Not Null): 계좌명
* `acc_company_nm` (String, Not Null): 금융사명
* `acc_order` (Integer, 기본값 1): 정렬 순서
* `dt_opened` (String, Null 허용): 계좌 개설일
* `cash_balance` (Integer, 기본값 0): 현금 잔고
* `dt_created` (DateTime): 생성 일시
* `dt_deleted` (DateTime, Null 허용): 삭제 일시 (소프트 삭제용)

## 2. 설계 의견 및 제약 사항
* **스키마 적합성**: 기획안에서 요구하는 모든 항목(계좌코드, 계좌명, 금융사명, 상태)을 기존 스키마로 충분히 수용할 수 있습니다. 상태(활성/비활성)는 `dt_deleted` 컬럼의 Null 여부로 판단하는 **소프트 삭제(Soft Delete)** 방식을 권장합니다.
* **무결성 유지**: `Transaction`, `TransactionCash`, `AccountDailyBalance` 테이블들이 `account.acc_cd`를 외래키(ForeignKey)로 참조하고 있습니다. 따라서 계좌 삭제 요청이 들어오더라도 `DELETE` 쿼리 대신 `UPDATE account SET dt_deleted = CURRENT_TIMESTAMP WHERE acc_cd = ?` 방식으로 처리하여 무결성 오류를 방지해야 합니다.
* **초기 데이터 적재**: 기존에 코드 레벨이나 마이그레이션 스크립트로 들어갔던 계좌 데이터가 있다면, 이와 충돌하지 않도록 유의해야 합니다.

## 3. 결론
* DB 스키마는 **변경 없이 기존 `account` 테이블을 그대로 사용**해도 무방합니다. BE에서 CRUD를 구현할 때 소프트 삭제 로직만 신경 써주시면 됩니다.
