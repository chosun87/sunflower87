## 1. 개요 및 배경
최근 백엔드 서버 구동 과정에서 **KRX 주식 마스터 동적 시딩 에러**, **SQLite 데이터베이스 파일 잠금(Lock)**, **서버 구동 시 중복 로그인(2회)** 등의 연쇄적인 장애와 최적화 이슈가 식별되었습니다. 

이를 근본적으로 해결하기 위해 백엔드 데이터베이스 마이그레이션 구조를 재정비하고, 데이터 자동 보정(Self-healing) 메커니즘을 도입하였으며, FastAPI의 생명 주기 관리자(`lifespan`)를 적용해 불필요한 리소스 낭비를 없애는 최적화 작업을 완료하였습니다. 본 문서에서는 상세 문제 진단 내용과 조치 사항을 공유해 드립니다.

---

## 2. 주요 문제 진단 및 조치 사항 (3대 요약)

### 🚀 [조치 1] DB 마이그레이션(Schema Evolution) 안정화 및 자동화
* **기존 문제:** 
  1. `migrate.py` 내의 이모지 문자(🚀, ✅ 등)가 윈도우 기본 인코딩 환경(`cp949`)에서 실행될 때 **`UnicodeEncodeError`**를 유발하여 마이그레이션이 실패함.
  2. 서버 구동 진입점(`main.py`)에서 마이그레이션 자동 실행 코드가 누락되어 신규 컬럼(`market`, `is_active`)이 없는 상태로 DB 쿼리가 시도되면서 서비스 중단을 유발함.
  3. `migrate.py` 내에 `conn.commit()`이 누락되어 실제 스키마 변경 사항이 반영되지 않았음.
* **조치 사항:** 
  * `migrate.py` 내의 이모지를 안전한 텍스트 포맷으로 변경하고, 누락된 `conn.commit()`을 복원하여 실행 무결성을 확보했습니다.
  * 백엔드 기동 프로세스에 마이그레이션 단계를 강제로 결합하여, DB 파일이 존재하면 서버 구동 직전에 항상 스키마 점진적 진화(ALTER TABLE)를 안전하게 먼저 수행하도록 수정했습니다.

### 💊 [조치 2] SQLAlchemy 세션 롤백 안정화 및 기존 데이터 자가 보정 (Self-healing)
* **기존 문제:** 
  1. `database.py` 내의 7일간의 영업일 탐색 루프 도중 한 번이라도 DB 에러가 발생하면 SQLAlchemy 세션이 무효화되어, 이후 루프 회차와 오프라인 백업 시딩 단계까지 연쇄적으로 쿼리 오류 및 `UNIQUE constraint` 실패를 유발함.
  2. 스키마가 업데이트되었으나, 이전에 생성되어 테이블 내에 이미 저장되어 있던 기존 종목 데이터들의 `market` 컬럼 값이 비어(`None`) 있는 상태로 머물러 있어 데이터 정합성이 깨짐.
* **조치 사항:** 
  * 시딩 루프의 예외 처리 블록 내에 `db.rollback()`을 명시적으로 추가하여 개별 날짜별 조회 오류가 나더라도 세션 무결성을 온전히 유지하도록 고쳤습니다.
  * 온라인/오프라인 시딩 시 기존 캐시에 `market` 컬럼 값이 비어있는 레코드가 발견될 경우, 적합한 시장 분류(KOSPI, KOSDAQ, ETF)로 **자동 보정(Self-healing)**하여 업데이트해 주는 하이브리드 로직을 추가 적용했습니다. (총 170여 개의 종목 데이터 정제 완료)

### ⚡ [조치 3] FastAPI Lifespan 기반의 중복 로그인(2회) 및 리소스 낭비 제거
* **기존 문제:** 
  * Uvicorn의 개발 환경 실시간 감시 옵션(`reload=True`) 특성으로 인해, 파일 변경 시 부모 프로세스와 실제 요청을 처리하는 자식 프로세스가 모두 `main.py` 전역 영역의 `init_db()` 코드를 로드하게 됨. 이로 인해 서버 구동 시마다 KRX 로그인이 불필요하게 **2회 중복 수행**되어 기동 시간이 느려지고 계정 세션 낭비가 발생함.
* **조치 사항:** 
  * `main.py`에 FastAPI 공식 표준 권장 명세인 **`lifespan` 비동기 컨텍스트 매니저**를 구현했습니다.
  * 탑레벨 전역 공간에 노출되어 있던 `run_migrations()`와 `init_db()`를 `lifespan` 블록 안으로 온전히 격리시켰습니다.
  * 결과적으로, 파일 변경이나 임포트 감시 단계에서는 로그인 작업이 차단되고, **실제 활성화되는 웹 서비스 워커 프로세스 구동 시점에 딱 1회만 기동**되도록 최적화를 완료했습니다.

## 3. 프로덕션 운영 관점의 아키텍처 제언 (무니 님 참고 사항)
현재 적용된 최적화 구조는 **로컬 단일 워커 개발 단계에서는 리소스 소모를 극적으로 줄여주는 베스트 프랙티스**입니다. 다만 향후 대용량 처리를 위해 상용 환경(Production)에서 멀티 프로세스를 도입할 경우 다음 사항을 염두에 두어야 합니다.

> [!WARNING]
> **멀티 워커(Multi-worker) 환경 구동 시 SQLite 파일 락 위험**
> Uvicorn 기동 옵션에 `--workers 4` 등을 지정하여 멀티 프로세스로 띄울 경우, 각 워커의 `lifespan`이 거의 동시에 동작하면서 SQLite 파일에 쓰기(ALTER TABLE, INSERT)를 시도하게 됩니다. 이 경우 다중 프로세스 간 자원 점유 경합으로 인해 다시 한번 `database is locked`가 발생할 수 있습니다.
> 
> **권장 대응 방식 (상용 환경 배포 시):**
> 실제 운영 서버 배포 단계(CI/CD 파이프라인이나 `run.ps1` 가동 전 단계)에서는 `lifespan` 내부의 마이그레이션 연동을 분리하고, **`python migrate.py`를 단 1회 선행 가동하여 스키마를 완벽히 확보한 후에 웹 프로세스군을 올리는 독립적 아키텍처**로 격리하여 구성하는 것이 아키텍처상 극도로 안전합니다.
