## 📌 [COMMON] 공통 요구사항
- **시계열 데이터베이스화:** 장중 실시간 스크래핑 대신, 최근 매매거래일 기준 60일치의 시고저종(OHLCV) 주가 데이터를 로컬 DB에 누적 관리하여 향후 기술적 분석의 기반을 마련한다.
- **증분 수집 제어:** 이미 과거 데이터가 확보된 종목은 중복 호출을 피하고 전일자 데이터만 컴팩트하게 추가 적재하는 효율적 증분 동기화 알고리즘을 구현한다.
- **평가 기준 통일:** 보유 종목의 '현재가'는 최근 매매거래일의 **'전일 종가'**를 기준으로 산정하여 전체 자산을 정산한다.

## 🌍 [BE_TASK] 백엔드 상세 구현 지침

### 1. 작업 디렉토리 및 전역 변수 통제
- 작업 공간: `be/`
- `main.py` 또는 구성 파일 최상단에 확장 기간 제어용 전역 변수를 반드시 선언하라.
  `TRADE_DATE_PERIOD = 60` (필요시 운영 중 즉각 변경 가능하도록 바인딩)

### 2. SQLite 데이터베이스 시계열 캐시 스키마 신설 (`stock_ohlcv_cache`)
종목별 일자별 주가 데이터를 누적 적재할 테이블을 신설하라.
- **`stock_ohlcv_cache` 테이블 정의:**
  - `stock_code`: String (종목/ETF 6자리 고유 코드)
  - `trade_date`: String (매매일 - 포맷: 'YYYYMMDD')
  - `open_price`: Integer (시가)
  - `high_price`: Integer (고가)
  - `low_price`: Integer (저가)
  - `close_price`: Integer (종가 -> ★자산 평가의 '현재가'로 활용)
  - `volume`: Integer (거래량)
  - **Composite Primary Key:** `(stock_code, trade_date)` 구조로 설정하여 동일 종목의 동일 날짜 데이터 중복 적재를 원천 차단하라.

### 3. OHLCV 증분 수집 알고리즘 구현
보유 자산 조회 프로세스 가동 시, 각 보유 종목(`stocks` 테이블 기준)에 대해 내부 팩토리 펑션을 트리거하여 캐시를 동기화하라. `pykrx.stock.get_market_ohlcv_by_date` API를 활용한다.

*   **케이스 1) 신규 등록된 종목 (로컬 DB 내 과거 기록이 없는 경우):**
    - 조회 시작일(`전일자 - TRADE_DATE_PERIOD + 1일`)부터 `전일자`까지를 파라미터로 지정하여 **총 60일치**의 OHLCV 데이터를 한 번에 가져와 `stock_ohlcv_cache` 테이블에 통째로 `INSERT` 하라.
*   **케이스 2) 기 등록된 종목 (이미 로컬 DB에 기록이 누적되어 있는 경우):**
    - DB 내 해당 종목의 최신 `trade_date`를 조회하라. 이미 60일치 이상이 꽉 채워져 있다면, 국동 부하를 막기 위해 **오직 `전일자` 단 1일치의 데이터만** `pykrx`로 획득하여 테이블에 추가(Append)하라.

### 4. 보유 자산 조회 API (`GET /api/accounts`) 사양 확장
- `stocks` 테이블의 현재고 리스트를 반환할 때, `stock_ohlcv_cache` 테이블에서 **가장 최신 날짜(즉, 전일자)의 `close_price`**를 조회하여 매핑하라.
- 이 `close_price`를 JSON 응답의 `current_price` 필드로 매핑하여 프런트엔드에 리턴하라.
