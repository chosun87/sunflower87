## 📌 [COMMON] 공통 요구사항
- **네이밍 컨벤션 표준화 준수:** 시스템 전체의 계좌 식별자 명칭은 백엔드 데이터베이스 스키마 규격에 맞추어 **`acc_cd`**로 전면 통일한다. 기존에 혼용되던 `acc_code` 또는 `account_number` 명칭은 전면 폐기한다.
- **예수금 뷰 신설:** 주식 자산 외에 계좌별 현금 잔고가 대시보드에 명확히 표기되도록 UI를 확장한다.

## 🌍 [FE_TASK] 프런트엔드 상세 구현 지침

### 1. 작업 디렉토리: `fe/`

### 2. 주식계좌 선택 Dropdown 및 예수금(Cash Balance) UI 배치
- **Dropdown 세팅:** `GET /api/accounts` API와 연동하되, 화면 노출은 `optionLabel="acc_nm"`, 선택 시 내부 데이터 매핑은 **`optionValue="acc_cd"`**로 명확히 바인딩하라.
- **예수금 뷰 추가:** [탭 1: 보유 자산 상세] 영역의 `DataTable` 상단에 컴팩트한 배너 또는 카드 컴포넌트를 배치하여, 백엔드 응답 데이터의 **`cash_balance`** 값이 원화 포맷(`toLocaleString()`)으로 실시간 노출되도록 UI를 구현하라.

### 3. ★ 백엔드 통일 규격에 맞춘 매매 등록 API 송출 프로토콜
- 사용자가 폼 입력을 마치고 [등록] 버튼을 누를 때, 데이터베이스 외래키 정합성을 위해 **계좌 식별 파라미터명을 반드시 `acc_cd`로 정의**하여 `POST /api/transactions/add` 바디에 실어 보내라.
  - *전송 JSON Body 최종 규격:* `{ "acc_cd": selectedAccCd, "type": "BUY", "code": "005930", ... }`
- **제출 프로세스 로딩 스핀 확장:** 신규 종목 매수 시 백엔드 내부의 60일치 시계열 수집 연산으로 인해 응답이 지연될 수 있으므로, 요청 송출 순간 폼 전체를 `disabled` 처리하거나 등록 버튼에 `loading={isSubmitting}` 상태를 바인딩하여 중복 클릭을 원천 차단하라.

### 4. 보유 자산 상세 DataTable 연산 및 색상 반전 규칙 (R8 규격 승계)
- **[현재가] 컬럼:** 백엔드가 제공하는 전일 마감 종가(`current_price`) 출력.
- **[평가금액] 컬럼:** `quantity` × `current_price`
- **[수익률] 컬럼:** `((current_price - avg_price) / avg_price) * 100` 
  - 연산 결과가 양수(+)이면 **Red 톤 텍스트**, 음수(-)이면 **Blue 톤 텍스트** 스타일(CSS)을 동적으로 입혀 가시성을 보장하라. (.toFixed(2) 제한)

### 5. 전면적 데이터 리프레시 (Refetch)
- 매매 기록의 등록, 수정, 삭제가 성공하여 팝업 모달이 닫히는 즉시 상위 대시보드의 `fetchAccountData()`를 강제 트리거하여, 자산 상세 테이블, 예수금 배너, 하단 매매 히스토리 테이블이 최신 SQLite DB 상태로 일제히 리프레시되도록 처리하라.
